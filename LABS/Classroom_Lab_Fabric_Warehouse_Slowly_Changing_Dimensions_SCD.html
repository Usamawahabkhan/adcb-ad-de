Business Scenario

A retail company receives periodic customer snapshots from an operational customer system. Customer attributes can change over time. The data warehouse team must decide how to store those changes for reporting.

Batch 1 - initial source extract Customer 1001: New York, Silver Customer 1002: Los Angeles, Gold Customer 1003: Chicago, Bronze Batch 2 - later source extract Customer 1001: Boston, Gold -- city and loyalty changed Customer 1002: new email address -- email changed Customer 1003: no change Customer 1004: new customer

Learning Objectives

  • Understand what Slowly Changing Dimensions are and why they matter in data warehousing.
  • Compare SCD Type 0, Type 1, Type 2, Type 3, Type 4, and Type 6.
  • Create sample customer snapshots in Microsoft Fabric Warehouse.
  • Apply each SCD pattern using Fabric-friendly SQL.
  • Understand how current-state and history-preserving dimensions behave differently.

Basic Definitions for Each SCD Type

SCD typePatternDefinitionExample use case
SCD Type 0Fixed attributeThe original value is preserved and not changed by later source updates.Original signup city, original birth date, first purchase date.
SCD Type 1OverwriteThe dimension row is updated in place and history is not preserved.Correcting misspelled names, latest email address, latest phone number.
SCD Type 2Full historyA new row is inserted for each meaningful change. Old rows are expired with effective dates and current flag.Customer address history, loyalty tier history, product category history.
SCD Type 3Limited historyOne or a small number of previous values are stored in extra columns on the same row.Current city plus previous city.
SCD Type 4Separate history tableThe current dimension table stores only current values. A separate history table stores older values.High-volume audit history where current queries must stay simple.
SCD Type 6HybridCombines Type 1, Type 2, and Type 3 ideas, commonly using historical rows plus previous/current attributes.Advanced dimensions where both history and current/previous values are useful.

Microsoft Fabric Warehouse Compatibility Notes

  • Use VARCHAR instead of NVARCHAR.
  • Do not use IDENTITY in this beginner classroom lab.
  • Do not use DEFAULT constraints in CREATE TABLE.
  • Use explicit inserts and explicit surrogate keys generated with ROW_NUMBER().
  • Use lab-owned tables only. These scripts do not modify sample Warehouse tables.

Step 1: Create source snapshot table and load Batch 1

Objective: Create a source-like customer snapshot table and load the first extract.

Script for this step

CREATE TABLE dbo.stage_customer_snapshot (
    BatchID             INT          NOT NULL,
    BusinessCustomerID  INT          NOT NULL,
    FullName            VARCHAR(200) NOT NULL,
    Email               VARCHAR(200) NOT NULL,
    City                VARCHAR(100) NOT NULL,
    LoyaltyTier         VARCHAR(50)  NOT NULL,
    SourceEffectiveDate DATE         NOT NULL

-- Batch 1: Initial customer extract from the source system

(1, 1001, 'James Wilson',    'james.wilson@example.com',    'New York',    'Silver', '2024-01-01'),
(1, 1002, 'Sophia Martinez', 'sophia.martinez@example.com', 'Los Angeles', 'Gold',   '2024-01-01'),
(1, 1003, 'Ethan Brown',     'ethan.brown@example.com',     'Chicago',     'Bronze', '2024-01-01');

SELECT *
FROM dbo.stage_customer_snapshot

Step 2: SCD Type 0 initial load

Objective: Create a fixed-value dimension where selected attributes never change.

Script for this step

CREATE TABLE dbo.DimCustomer_Type0 (
    CustomerKey         INT          NOT NULL,
    BusinessCustomerID  INT          NOT NULL,
    FullName            VARCHAR(200) NOT NULL,
    OriginalCity        VARCHAR(100) NOT NULL,
    OriginalLoyaltyTier VARCHAR(50)  NOT NULL,
    CreatedDate         DATE         NOT NULL
);

INSERT INTO dbo.DimCustomer_Type0
SELECT
    ROW_NUMBER() OVER (ORDER BY BusinessCustomerID) AS CustomerKey,
    BusinessCustomerID,
    FullName,
    City AS OriginalCity,
    LoyaltyTier AS OriginalLoyaltyTier,
    SourceEffectiveDate AS CreatedDate
FROM dbo.stage_customer_snapshot
WHERE BatchID = 1;

SELECT *
FROM dbo.DimCustomer_Type0
ORDER BY BusinessCustomerID;

Expected result

Three Type 0 rows are inserted with original city and original loyalty tier.

Step 9: Apply SCD Type 0

Objective: Preserve original values and insert only new customers.

Script for this step

-- SCD Type 0 behavior:
-- Do nothing to existing rows.
-- Only insert brand-new customers if the dimension did not already know them.
INSERT INTO dbo.DimCustomer_Type0
SELECT
    (SELECT COALESCE(MAX(CustomerKey), 0) FROM dbo.DimCustomer_Type0)
        + ROW_NUMBER() OVER (ORDER BY s.BusinessCustomerID) AS CustomerKey,
    s.BusinessCustomerID,
    s.FullName,
    s.City AS OriginalCity,
    s.LoyaltyTier AS OriginalLoyaltyTier,
    s.SourceEffectiveDate AS CreatedDate
FROM dbo.stage_customer_snapshot AS s
LEFT JOIN dbo.DimCustomer_Type0 AS d
    ON s.BusinessCustomerID = d.BusinessCustomerID
WHERE s.BatchID = 2
  AND d.BusinessCustomerID IS NULL;

SELECT *
FROM dbo.DimCustomer_Type0
ORDER BY BusinessCustomerID;

Expected result

Customer 1001 keeps OriginalCity = New York; customer 1004 is inserted.

Step 10: Apply SCD Type 1

Objective: Overwrite changed rows and insert new rows.

Script for this step

-- SCD Type 1 behavior:
-- Overwrite changed attributes. No history is preserved.
UPDATE d
SET
    d.FullName = s.FullName,
    d.Email = s.Email,
    d.City = s.City,
    d.LoyaltyTier = s.LoyaltyTier,
    d.LastUpdatedDate = s.SourceEffectiveDate
FROM dbo.DimCustomer_Type1 AS d
JOIN dbo.stage_customer_snapshot AS s
    ON d.BusinessCustomerID = s.BusinessCustomerID
WHERE s.BatchID = 2
  AND (
        d.FullName <> s.FullName
     OR d.Email <> s.Email
     OR d.City <> s.City
     OR d.LoyaltyTier <> s.LoyaltyTier
  );

-- Insert new customers.
INSERT INTO dbo.DimCustomer_Type1
SELECT
    (SELECT COALESCE(MAX(CustomerKey), 0) FROM dbo.DimCustomer_Type1)
        + ROW_NUMBER() OVER (ORDER BY s.BusinessCustomerID) AS CustomerKey,
    s.BusinessCustomerID,
    s.FullName,
    s.Email,
    s.City,
    s.LoyaltyTier,
    s.SourceEffectiveDate AS LastUpdatedDate
FROM dbo.stage_customer_snapshot AS s
LEFT JOIN dbo.DimCustomer_Type1 AS d
    ON s.BusinessCustomerID = d.BusinessCustomerID
WHERE s.BatchID = 2
  AND d.BusinessCustomerID IS NULL;

SELECT *
FROM dbo.DimCustomer_Type1
ORDER BY BusinessCustomerID;

Expected result

Customer 1001 now shows Boston and Gold; customer 1002 has the new email; no history is preserved.

Step 11: Apply SCD Type 2

Objective: Expire old current rows and insert new current versions.

Script for this step

-- SCD Type 2 behavior:
-- Expire the current row when tracked attributes change.
-- Then insert a new current row.

-- 1) Expire changed current records.
UPDATE d
SET
    d.EffectiveEndDate = DATEADD(DAY, -1, s.SourceEffectiveDate),
    d.IsCurrent = 0
FROM dbo.DimCustomer_Type2 AS d
JOIN dbo.stage_customer_snapshot AS s
    ON d.BusinessCustomerID = s.BusinessCustomerID
WHERE s.BatchID = 2
  AND d.IsCurrent = 1
  AND (
        d.FullName <> s.FullName
     OR d.Email <> s.Email
     OR d.City <> s.City
     OR d.LoyaltyTier <> s.LoyaltyTier
  );

-- 2) Insert new current records for changed or new customers.
INSERT INTO dbo.DimCustomer_Type2
SELECT
    (SELECT COALESCE(MAX(CustomerKey), 0) FROM dbo.DimCustomer_Type2)
        + ROW_NUMBER() OVER (ORDER BY s.BusinessCustomerID) AS CustomerKey,
    s.BusinessCustomerID,
    s.FullName,
    s.Email,
    s.City,
    s.LoyaltyTier,
    s.SourceEffectiveDate AS EffectiveStartDate,
    CAST(NULL AS DATE) AS EffectiveEndDate,
    1 AS IsCurrent
FROM dbo.stage_customer_snapshot AS s
LEFT JOIN dbo.DimCustomer_Type2 AS current_d
    ON s.BusinessCustomerID = current_d.BusinessCustomerID
   AND current_d.IsCurrent = 1
WHERE s.BatchID = 2
  AND current_d.BusinessCustomerID IS NULL;

SELECT *
FROM dbo.DimCustomer_Type2
ORDER BY BusinessCustomerID, EffectiveStartDate;

Expected result

Changed customers have old rows expired and new current rows inserted; customer 1004 is inserted.

Step 13: Apply SCD Type 4

Objective: Store current values separately from history records.

Script for this step

-- SCD Type 4 behavior:
-- Current values are stored in one table.
-- Historical before-images are stored in a separate history table.

-- 1) Capture old current values into the history table before overwriting.
INSERT INTO dbo.DimCustomer_Type4_History
SELECT
    (SELECT COALESCE(MAX(HistoryKey), 0) FROM dbo.DimCustomer_Type4_History)
        + ROW_NUMBER() OVER (ORDER BY d.BusinessCustomerID) AS HistoryKey,
    d.BusinessCustomerID,
    d.FullName,
    d.Email,
    d.City,
    d.LoyaltyTier,
    'Changed in source batch 2' AS HistoryReason,
    s.SourceEffectiveDate AS HistoryCapturedDate
FROM dbo.DimCustomer_Type4_Current AS d
JOIN dbo.stage_customer_snapshot AS s
    ON d.BusinessCustomerID = s.BusinessCustomerID
WHERE s.BatchID = 2
  AND (
        d.FullName <> s.FullName
     OR d.Email <> s.Email
     OR d.City <> s.City
     OR d.LoyaltyTier <> s.LoyaltyTier
  );

-- 2) Overwrite current table.
UPDATE d
SET
    d.FullName = s.FullName,
    d.Email = s.Email,
    d.City = s.City,
    d.LoyaltyTier = s.LoyaltyTier,
    d.LastUpdatedDate = s.SourceEffectiveDate
FROM dbo.DimCustomer_Type4_Current AS d
JOIN dbo.stage_customer_snapshot AS s
    ON d.BusinessCustomerID = s.BusinessCustomerID
WHERE s.BatchID = 2
  AND (
        d.FullName <> s.FullName
     OR d.Email <> s.Email
     OR d.City <> s.City
     OR d.LoyaltyTier <> s.LoyaltyTier
  );

-- 3) Insert new customers into current table.
INSERT INTO dbo.DimCustomer_Type4_Current
SELECT
    (SELECT COALESCE(MAX(CustomerKey), 0) FROM dbo.DimCustomer_Type4_Current)
        + ROW_NUMBER() OVER (ORDER BY s.BusinessCustomerID) AS CustomerKey,
    s.BusinessCustomerID,
    s.FullName,
    s.Email,
    s.City,
    s.LoyaltyTier,
    s.SourceEffectiveDate AS LastUpdatedDate
FROM dbo.stage_customer_snapshot AS s
LEFT JOIN dbo.DimCustomer_Type4_Current AS d
    ON s.BusinessCustomerID = d.BusinessCustomerID
WHERE s.BatchID = 2
  AND d.BusinessCustomerID IS NULL;

SELECT * FROM dbo.DimCustomer_Type4_Current ORDER BY BusinessCustomerID;
SELECT * FROM dbo.DimCustomer_Type4_History ORDER BY BusinessCustomerID, HistoryKey;

Expected result

Current rows are overwritten and changed before-images are written to the history table.

Step 14: Apply SCD Type 6

Objective: Use a hybrid SCD pattern with Type 2 dates and previous city.

Script for this step

-- SCD Type 6 behavior:
-- Hybrid example: Type 2 history + Type 3 previous city.

-- 1) Expire changed current records.
UPDATE d
SET
    d.EffectiveEndDate = DATEADD(DAY, -1, s.SourceEffectiveDate),
    d.IsCurrent = 0
FROM dbo.DimCustomer_Type6 AS d
JOIN dbo.stage_customer_snapshot AS s
    ON d.BusinessCustomerID = s.BusinessCustomerID
WHERE s.BatchID = 2
  AND d.IsCurrent = 1
  AND (
        d.FullName <> s.FullName
     OR d.Email <> s.Email
     OR d.City <> s.City
     OR d.LoyaltyTier <> s.LoyaltyTier
  );

-- 2) Insert changed or new current records.
-- PreviousCity is taken from the latest expired row when available.
INSERT INTO dbo.DimCustomer_Type6
SELECT
    (SELECT COALESCE(MAX(CustomerKey), 0) FROM dbo.DimCustomer_Type6)
        + ROW_NUMBER() OVER (ORDER BY s.BusinessCustomerID) AS CustomerKey,
    s.BusinessCustomerID,
    s.FullName,
    s.Email,
    s.City,
    expired.City AS PreviousCity,
    s.LoyaltyTier,
    s.SourceEffectiveDate AS EffectiveStartDate,
    CAST(NULL AS DATE) AS EffectiveEndDate,
    1 AS IsCurrent
FROM dbo.stage_customer_snapshot AS s
LEFT JOIN dbo.DimCustomer_Type6 AS current_d
    ON s.BusinessCustomerID = current_d.BusinessCustomerID
   AND current_d.IsCurrent = 1
LEFT JOIN dbo.DimCustomer_Type6 AS expired
    ON s.BusinessCustomerID = expired.BusinessCustomerID
   AND expired.IsCurrent = 0
   AND expired.EffectiveEndDate = DATEADD(DAY, -1, s.SourceEffectiveDate)
WHERE s.BatchID = 2
  AND current_d.BusinessCustomerID IS NULL;

SELECT *
FROM dbo.DimCustomer_Type6
ORDER BY BusinessCustomerID, EffectiveStartDate;

Expected result

Customer 1001 has history rows and the new current row includes PreviousCity = New York.

Discussion Questions

  • Which SCD type is best when the business only cares about the latest customer email?
  • Which SCD type is best when analysts need to know what city a customer belonged to at the time of a sale?
  • Why does Type 2 create more rows than Type 1?
  • Why might Type 4 be useful when current queries need to stay fast and simple?
  • What extra complexity does Type 6 introduce?

Expected Learning Results

  • Type 0 preserves original values.
  • Type 1 overwrites history.
  • Type 2 creates multiple rows for changed customers.
  • Type 3 stores one previous value in the current row.
  • Type 4 separates current data and history data.
  • Type 6 combines full history with a previous-value column.
